# CLAUDE.md — Инструкции за AI асистента

## Език и тон

Работим на български по подразбиране и си говорим на „ти". Тонът трябва да е
директен, кратък и практичен, без излишно разтягане. Ако не поискам повече
детайл, държи отговорите стегнати.

## При неясни изисквания

Не си доизмисляй — питай. Ако не си достатъчно сигурен, по-добре изясни,
вместо да гадаеш. При по-съществени промени опиши какво смяташ да направиш
преди да го направиш.

## За проекта

- **Какво е:** Локален уеб дашборд (`dashboard.py`) за мониторинг на AI CLI
  сесии — Claude Code, OpenClaw, Codex CLI, GitHub Copilot CLI.
- **Стек:** Чист Python 3.8+ (само stdlib + sqlite3). Без frameworks, без
  npm, без external dependencies.
- **Стартиране:** `python3 dashboard.py` → http://localhost:7788
- **Порт:** По подразбиране 7788, може да се смени с `PORT` env variable.

## Архитектура (накратко)

```
dashboard.py
├── PANELS[]              # Дефиниции на панелите (key, title, color, install)
├── get_sessions(key)     # Диспечер — извиква правилния collector
├── _get_openclaw()       # OpenClaw: `openclaw sessions --json`
├── _get_claude()         # Claude Code: ~/.claude/projects/*/*.jsonl
├── _get_codex()          # Codex CLI: ~/.codex/state_5.sqlite (threads таблица)
├── _get_copilot()        # Copilot CLI: ~/.copilot/session-state/*/workspace.yaml
├── _claude_rate_limits() # Claude: `claude -p` → rate_limit_event
├── _codex_rate_limits()  # Codex: app-server JSON-RPC → WebSocket
├── _session()            # Единен формат за всички сесии
├── _build_panel()        # Единен HTML шаблон за всички панели
├── build_html(force_rl)  # Сглобява цялата страница
├── delete_session()      # POST /delete
├── launch_session()      # POST /launch
└── Handler (HTTP server) # GET / + POST /delete, /launch
```

### Единен формат на сесия

Всеки collector връща dict `{"ok": bool, "sessions": [...], "rate_limits": {...}}`.
Сесиите са списък от dict-ове:

```python
{"id", "full_id", "cwd", "cwd_full", "model", "age", "age_s", "tokens", "ctx_tokens", "last_ts", "last_sort"}
```

### Къде стоят данните

| CLI              | Път                                          | Формат          |
|------------------|----------------------------------------------|-----------------|
| Claude Code      | `~/.claude/projects/*/*.jsonl`               | JSONL            |
| OpenClaw         | команда `openclaw sessions --json`           | JSON (stdout)    |
| Codex CLI        | `~/.codex/state_5.sqlite` → таблица `threads`| SQLite           |
| Copilot CLI      | `~/.copilot/session-state/*/workspace.yaml`  | YAML + JSONL     |

### Rate limits

| CLI         | Метод                                              | Кеш                              |
|-------------|----------------------------------------------------|------------------------------------|
| Claude Code | `claude -p ok --no-session-persistence` → stream-json → `rate_limit_event` | `~/.claude/.rate_limits_cache.json` |
| Codex CLI   | `codex app-server` → WebSocket JSON-RPC → `account/rateLimits/read` | `~/.codex/.rate_limits_cache.json`  |

- **Нормален refresh** (auto, на 30 сек): чете от кеш файл — бързо (~0.5 сек)
- **„Обнови" бутон** (`/?force=1`): вика API-тата за свежи данни (~5-8 сек)
- Claude дава само `status` + `resetsAt` (без процент при нисък usage)
- Codex дава процент (usedPercent) + resetsAt за 5ч и седмичен лимит

### Context windows

`CTX_WINDOWS` dict-ът в началото на файла определя размера на контекстния
прозорец по модел. Ако добавяш нов модел с различен context — добави го там.

### Функционалности на UI

- **Търсене/филтриране:** Глобален search input — филтрира редове във всички панели
- **Сортиране по колони:** Клик на заглавие → asc/desc, `data-v` атрибути за правилна сортировка
- **Терминал бутон (⌨):** Отваря Terminal.app (macOS) или gnome-terminal (Linux) със сесията
- **Изтриване (✕):** Трие сесията (Claude/Codex/Copilot — файлове; OpenClaw — API)
- **Rate limit ленти:** В header-а на Claude и Codex панелите

## Правила при промени

- **Един файл:** Всичко е в `dashboard.py`. Не разбивай на модули без причина.
- **Без external deps:** Не добавяй pip/npm пакети. stdlib е достатъчен.
- **Единен шаблон:** Всички панели минават през `_build_panel()`. Не прави
  custom HTML за отделен панел.
- **Тествай преди commit:** `python3 -c "import py_compile; py_compile.compile('dashboard.py', doraise=True)"`,
  после стартирай и провери http://localhost:7788.
- **Български UI:** Всички надписи в интерфейса са на български.
- **macOS:** Терминалите се отварят с `osascript` (AppleScript). `_IS_MAC` флаг.

## Поведенчески насоки (за по-малко грешки)

**Компромис:** Тези правила натежават към внимателност пред скорост. За тривиални
задачи — преценявай сам.

### 1. Мисли преди да пишеш код

**Не допускай. Не крий объркване. Изнасяй компромисите.**

- Заяви допусканията си изрично. Ако не си сигурен — питай.
- Ако има няколко интерпретации — покажи ги, не избирай мълчаливо.
- Ако по-прост подход съществува — кажи го. Отблъсквай когато е уместно.
- Ако нещо е неясно — спри. Назови какво те обърква. Питай.

### 2. Простота на първо място

**Минимум код, който решава проблема. Нищо спекулативно.**

- Без функционалности отвъд поисканото.
- Без абстракции за еднократен код.
- Без „гъвкавост" или „конфигурируемост", която не е поискана.
- Без error handling за невъзможни сценарии.
- Ако напишеш 200 реда, а може с 50 — пренапиши.

Тест: „Старши инженер би ли казал, че е прекалено сложно?" Ако да — опрости.

### 3. Хирургически промени

**Пипай само каквото трябва. Чисти само собствените си боклуци.**

При редакция на съществуващ код:
- Не „подобрявай" съседен код, коментари или форматиране.
- Не рефакторирай неща, които не са счупени.
- Спазвай съществуващия стил, дори да би го направил иначе.
- Ако забележиш несвързан мъртъв код — спомени го, не го трий.

Когато промените ти създадат сираци:
- Премахни import-и/променливи/функции, които ТВОИТЕ промени направиха неизползвани.
- Не премахвай предварително съществуващ мъртъв код, освен ако не е поискано.

Тест: Всеки променен ред трябва да се проследи директно до заявката на потребителя.

### 4. Целенасочено изпълнение

**Дефинирай критерии за успех. Повтаряй до потвърждение.**

Превръщай задачи във верифицируеми цели:
- „Добави валидация" → „Напиши тестове за невалидни входове, после ги накарай да минат"
- „Оправи бъга" → „Напиши тест, който го възпроизвежда, после го накарай да мине"
- „Рефакторирай X" → „Увери се, че тестовете минават преди и след"

За многостъпкови задачи изложи кратък план:
```
1. [Стъпка] → проверка: [какво]
2. [Стъпка] → проверка: [какво]
3. [Стъпка] → проверка: [какво]
```

Силни критерии за успех позволяват самостоятелна итерация. Слаби критерии
(„да работи") изискват постоянно уточняване.
